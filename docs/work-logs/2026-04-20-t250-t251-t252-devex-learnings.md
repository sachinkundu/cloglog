# Learnings — wt-devex-batch

**Date:** 2026-04-20

## What went well

- **Batching three tiny fixes into one PR saved three review cycles.** Zero file-level overlap among T-250/T-251/T-252, only CLAUDE.md touched by two. The alternative — three trivial PRs — would have burned three rounds of codex review, three CI passes, three merge waits. The batching rule of thumb that emerged: if the tasks share a theme (here, plugin/bootstrap devex), have no internal conflicts, and are each too small to warrant their own demo, one PR is strictly better.
- **Scope guard held against a tempting Boy Scout edit.** `.claude/agents/demo-reviewer.md:34` carries the same raw-`rev-parse` pattern as the templates T-251 was fixing. The devex worktree's AGENT_PROMPT explicitly scoped changes to `.cloglog/on-worktree-create.sh`, `plugins/cloglog/skills/demo/SKILL.md`, possibly another plugin template, and `CLAUDE.md`. The demo-reviewer agent template is outside the listed scope and — importantly — doesn't hit the slash-norm failure in practice because PR branches on this project are conventionally `wt-*` (no slashes). Noted for a future task, not silently smuggled in.
- **Codex reviewer's first pass caught both real problems.** Both findings (run-demo.sh half-fix, CLAUDE.md overclaim) were verifiable in ≤2 minutes of reading and were genuinely load-bearing. Round-2 `:pass:` was honest — the reviewer explicitly listed the files it re-read to verify the follow-up.

## Issues encountered

### Half-fix hazard when multiple scripts share a convention
- T-251's first cut fixed the template in `plugins/cloglog/skills/demo/SKILL.md` and was blind to the fact that `scripts/run-demo.sh` separately implements the *lookup* side of the same convention without normalization. The writer, the verifier (`check-demo.sh`), and the lookup (`run-demo.sh`) all needed to agree. Codex caught it on round 1.
- **Prevention:** when fixing a path-composition convention, grep for *every* consumer of the path, not just the producer. `grep -rn "FEATURE_NORM\|rev-parse --abbrev-ref HEAD\|docs/demos/" scripts/ plugins/` would have surfaced the triangle (writer + verifier + lookup) before the first push.

### Claiming a behavior that CI/bootstrap doesn't actually do
- T-252's first wording said `mcp-server/dist/` is "rebuilt by `on-worktree-create.sh` / CI". Both claims were false. `.cloglog/on-worktree-create.sh:27-29` only `npm install`s and only on `wt-mcp*` worktrees; `.github/workflows/ci.yml:68-96` runs `npm ci` + tests + contract-check with no build step. I had conflated "installs deps" with "rebuilds". The reviewer flagged this as [HIGH] because an engineer following the bad guidance would skip `cd mcp-server && make build` and then hit stale `dist/` at runtime (`.mcp.json` → `dist/index.js`; `scripts/demo-update-task-status.mjs` → `dist/server.js`).
- **Prevention:** before writing Environment Quirks prose that claims *any* component does X, `Read` the actual source of X. For "CI does Y" claims, that means reading `.github/workflows/*.yml`. For "bootstrap does Y" claims, read the bootstrap script top to bottom. The cost of two `Read` calls is well below the cost of a HIGH review finding plus a regeneration cycle.

### Showboat `exec` argv semantics
- `uvx showboat exec <file> <lang> [code]` — the `<lang>` positional determines the interpreter. For shell, pass `bash "…command…"`, not `bash -c "…"`. The templates in `plugins/cloglog/skills/demo/SKILL.md` show `bash -c '…'` under "exec quoting", which misreads what showboat expects — showboat then passes `-c` as the code and bash complains "option requires an argument". Took one retry to notice.
- **Fix candidate (not done in this PR — out of scope):** the "`exec` quoting" section of the demo SKILL.md may benefit from clarifying that showboat's two-positional form (`bash "…"`) is the canonical way to pass shell code to `exec`, and reserve `bash -c` only when shell features like pipes are genuinely needed. The existing examples use `bash "…uv run pytest… | grep…"` successfully, so the behavior is there — it just isn't called out in the quoting section.

## Suggestions for CLAUDE.md / plugin updates

- **`plugins/cloglog/skills/demo/SKILL.md` — `exec` quoting section:** document the `bash "<code>"` two-arg form as the primary way to run shell, with `bash -c` reserved for cases where you need `-c`-specific behavior. Several existing demos use the two-arg form successfully.
- **Codex reviewer prompt / plugin pattern:** the reviewer that caught the run-demo.sh half-fix explicitly listed the files it re-read (`scripts/run-demo.sh`, `scripts/check-demo.sh`) to verify the claim in the patch. That behavior is gold — it's what lets the reviewer catch claims about untouched files. Preserve it if the reviewer prompt is ever tightened.
- **This PR already implements** the T-247/T-249 fix candidates from prior learnings (uv sync --extra dev, demo-path normalization, dist/ wording). Future agents touching the same areas should treat those boxes as ticked.
